# EnvNexus 商业产品白皮书

> **文档版本**：v2.0
> **产品名称**：环枢 (EnvNexus)
> **产品定位**：AI 原生的智能环境治理引擎 — 自动诊断、计划式修复、主动发现

---

## 1. 产品愿景与定位

在现代企业 IT 环境中，终端用户的运行环境（操作系统、网络、依赖库、配置）日益复杂。传统的远程支持工具（如 TeamViewer、AnyDesk）虽然能解决问题，但赋予了支持人员**无限制的系统控制权**，带来了极大的安全合规风险。而传统的 IT 工单系统则依赖人工排查，效率低下、成本高昂。

**EnvNexus（环枢）** 致力于重塑企业 IT 支持与环境治理的边界。我们构建的不是一个简单的远程诊断工具，而是一个**智能环境治理引擎**——它能像经验丰富的 IT 专家一样理解问题、制定修复方案、安全执行，并且能**主动发现**环境中潜伏的隐患。

### 核心理念

**"用户只需描述问题，剩下的交给 EnvNexus。"**

- **智能诊断**：用户用自然语言描述问题（甚至只需粘贴一张报错截图），Agent 自动采集环境证据、推理根因。
- **计划式修复**：不是零散地执行命令，而是生成一份完整的**修复计划**——有步骤顺序、有风险标注、有回滚预案。用户一目了然地审批整个方案。
- **主动发现**：不等问题爆发，Agent 持续扫描环境健康状态，在问题萌芽阶段就发出预警并提供修复建议。
- **零信任安全**：默认只读、审批执行、全量审计。每一次系统变更都处于绝对的安全管控之下。

**一句话定义**：
`EnvNexus 是一个在零信任架构下，通过 AI 驱动实现终端环境智能诊断、计划式修复与主动治理的引擎。`

## 2. 核心痛点与解决方案

### 2.1 行业痛点

1. **信任危机**：传统远控软件权限过大，IT 人员可以随意浏览用户隐私文件或植入后门。
2. **排障成本高**：复杂的环境问题（如环境变量冲突、端口占用、依赖缺失）往往需要高级工程师耗费大量时间排查。
3. **修复不透明**：用户不知道 IT 人员在自己的电脑上做了什么，只能被动接受"修好了"的结论。
4. **合规审计缺失**：终端上的修复操作（如修改了哪个注册表、删除了哪个文件）通常没有结构化的操作日志，事后无法追责和回滚。
5. **被动响应**：传统 IT 支持只能在问题发生后介入，无法提前发现和预防环境隐患。
6. **SaaS 交付难**：不同企业客户对数据隐私要求不同，难以用单一的公有云产品满足所有企业的合规要求。

### 2.2 EnvNexus 解决方案

1. **智能诊断引擎**：内置多 Provider LLM 路由（支持 OpenAI、Anthropic、Gemini、DeepSeek、Ollama 等），自动分析系统日志、进程状态和网络配置。支持多模态输入——用户可以直接粘贴报错截图，Agent 自动提取关键信息并触发诊断。
2. **计划式修复**：诊断完成后，Agent 不是直接执行命令，而是生成一份结构化的**修复计划**（Remediation Plan）。计划中每一步都标注了风险等级、前置条件、验证方法和回滚策略。用户审批的是**整个方案**，而非零散的单条命令。
3. **分层审批机制**：
   - **L0 只读操作**：自动执行，无需审批（如查看进程列表、读取网络配置）。
   - **L1 低风险写操作**：计划审批后自动执行（如修改配置文件）。
   - **L2 中风险操作**：计划审批后，执行前再次确认（如重启服务、清理缓存）。
   - **L3 高风险操作**：计划审批 + 每步单独审批（如删除文件、修改注册表）。
4. **主动发现与智能巡检（Watchlist）**：Agent 不只是被动等待用户提问。用户可以用自然语言描述关注的指标（如"帮我盯着 MySQL 和 Redis，磁盘别超过 80%"），LLM 自动拆解为具体检测项（选择工具、设定阈值、推荐频率），用户确认后 Agent 持续巡检。同时内置通用规则包（DNS、磁盘、证书等）开箱即用，企业管理员也可通过平台下发合规策略。在问题萌芽阶段就发出预警，并自动生成修复建议。
5. **执行回滚**：每个修复步骤执行前自动记录状态快照。如果某步执行失败或验证不通过，自动按逆序回滚到变更前状态。
6. **全链路结构化审计**：从诊断发起、AI 推理、修复计划生成、用户审批到最终执行结果和验证，所有环节均生成结构化审计日志。
7. **OTA 静默更新**：Agent 自动检测并下载平台发布的新版本，管理员无需逐台推送。
8. **本地配置加密**：终端配置文件（含 API 密钥、平台连接信息）采用 AES-256-GCM 机器绑定加密，即使硬盘被非授权访问也无法读取明文。

## 3. 产品架构

EnvNexus 采用 **"云端大脑 + 智能客户端 + 强执行内核"** 的三位一体产品形态：

### 3.1 EnvNexus Platform (云端治理中枢)

*   **形态**：Web 控制台 (SaaS / 私有化部署)。
*   **功能**：
    *   **多租户管理**：支持多企业组织隔离。
    *   **策略配置中心 (Profiles)**：管理员在此配置 AI 模型接入（Model Profile）、安全拦截策略（Policy Profile）和客户端行为（Agent Profile）。
    *   **治理规则管理**：配置和下发环境治理规则包（网络、安全、性能、依赖等），终端 Agent 据此执行主动扫描。
    *   **健康态势仪表盘**：汇总所有终端的健康评分、漂移趋势、合规状态，一屏掌握全局。
    *   **分发管理**：一键生成租户专属的客户端下载链接（支持免填参数的静默激活）。
    *   **全局审计台**：查看所有终端的诊断记录、修复计划、审批流水和变更历史，支持 NDJSON 导出和脱敏。

### 3.2 EnvNexus Agent Desktop (终端交互界面)

*   **形态**：跨平台桌面应用程序（Windows / macOS / Linux），常驻系统托盘。
*   **功能**：
    *   **智能对话窗口**：用户可用自然语言描述问题，也可直接粘贴报错截图。Agent 自动理解意图并启动诊断。
    *   **修复计划审批台**：以清晰的步骤视图展示完整修复方案——每步的操作内容、风险等级、回滚策略一目了然。用户可以一键批准整个计划，也可以逐步审批高风险操作。
    *   **我的关注（Watchlist）**：用户用自然语言描述想要关注的指标（如"帮我盯着 MySQL 和磁盘"），Agent 自动拆解为具体检测项并请用户确认。确认后持续巡检，异常时推送告警和修复建议。
    *   **健康看板**：展示当前设备的健康评分、巡检状态、告警历史和趋势图。环境异常时通过系统通知主动提醒用户。
    *   **状态看板**：展示平台连接状态、Agent 版本和更新状态。
    *   *注：Desktop 仅为展示层，不包含任何提权或系统修改能力。*

### 3.3 EnvNexus Agent Core (智能治理内核)

*   **形态**：无 UI 的底层守护进程（Go 编写的独立二进制文件）。
*   **核心引擎**：
    *   **智能诊断引擎**：
        - **复杂度评估**：根据问题类型自动评估复杂度（Simple / Moderate / Complex / Critical），动态调整诊断深度和工具预算。
        - **分层证据收集**：先执行基础诊断工具，根据初步结果决定是否需要深入采集，避免不必要的系统开销。
        - **迭代推理**：如果首轮推理置信度不足，自动请求补充证据并进行多轮推理，直到达到可信结论。
    *   **修复计划生成器**：
        - 基于诊断结果，由 LLM 生成结构化修复计划（有序步骤 DAG）。
        - 引擎校验：验证所有工具名存在于注册表、强制使用注册表风险等级（不信任 LLM 评估）、校验依赖关系无环。
        - 自动注入回滚策略和验证步骤。
    *   **DAG 执行器**：
        - 按拓扑序执行修复步骤，每步执行前创建状态快照。
        - 执行后自动验证，失败时按逆序回滚。
        - 支持超时控制和中断恢复。
    *   **主动发现引擎（Watchlist）**：
        - **自然语言定义关注点**：用户用口语描述想监控的内容（如"帮我盯着 MySQL 和 Redis，磁盘别超过 80%"），LLM 自动拆解为结构化检测项（选择工具、设定阈值、推荐频率），用户确认后生效。LLM 还会主动补充建议（如用户关注 MySQL 连通性，额外建议监控进程资源占用）。
        - **四层规则来源**：用户自然语言定义（最灵活）、内置规则包（开箱即用）、平台下发规则（企业合规）、学习型规则（从历史修复中自动提取）。
        - **统一巡检调度**：所有规则来源统一注册为 WatchItem，按各自频率持续执行检测。
        - **健康评分系统**：综合多维指标计算设备健康分，评分下降时触发告警。
        - **闭环响应**：异常检测 → 告警通知 → 自动生成修复建议 → 用户审批 → 执行 → 验证恢复。
    *   **本地策略拦截网关**：即使云端 AI 建议执行某操作，如果本地策略禁止，Core 会直接在底层拦截，不予展示审批。
    *   **离线降级模式**：断网时仍可执行本地诊断、规则扫描和日志导出。内置规则和 Ollama 本地模型确保核心能力不依赖云端。
    *   **OTA 自更新**：自动向平台检查新版本，下载并替换自身二进制文件，支持 Windows 杀毒软件预热。
    *   **配置加密存储**：本地配置文件（`.enx`）使用 AES-256-GCM 加密，密钥从机器指纹派生。

*   **诊断工具生态**（33+ 内置工具）：

    | 类别 | 工具数 | 能力范围 |
    |------|--------|---------|
    | 网络诊断 | 10 | 网卡配置、代理检测、DNS 解析、Ping、端口扫描、路由表、HTTP 探测、TLS 证书检查、Traceroute |
    | 系统诊断 | 13 | 系统信息、磁盘、进程、环境变量、文件读取、已安装应用、事件日志、运行时依赖检查、Shell 执行 |
    | 容器诊断 | 3 | Docker 状态/日志/镜像、Docker Compose、Kubernetes 集群诊断 |
    | 数据库诊断 | 4 | MySQL、PostgreSQL、Redis、MongoDB 连通性与状态检查 |
    | 服务操作 | 2 | 服务重启、容器/进程重载 |
    | 缓存操作 | 1 | 缓存目录清理与重建 |

## 4. 核心业务场景 (Use Cases)

### 场景一：员工自助 IT 支持 — 智能诊断 + 计划式修复

1. 员工遇到开发环境报错，打开桌面端，粘贴报错截图并输入"这个怎么解决"。
2. Agent Core 通过 Vision 模型提取截图中的错误信息（如 `ECONNREFUSED 127.0.0.1:3306`），自动识别为 MySQL 连接问题。
3. 智能诊断引擎评估复杂度为 Moderate，分层采集证据：先检查进程列表和端口占用，发现 MySQL 服务未运行。
4. 修复计划生成器输出 3 步方案：
   - Step 1 [L0] 确认 MySQL 进程状态（只读，自动执行）
   - Step 2 [L2] 重启 MySQL 服务（中风险，需确认）
   - Step 3 [L0] 验证连接恢复（只读，自动执行）
5. 员工在桌面端看到完整修复计划，点击"批准"。Step 1 自动执行，Step 2 弹出确认框，员工再次确认后执行，Step 3 自动验证连接恢复。
6. 全过程无需 IT 人员介入，审计日志完整记录。

### 场景二：远程专家协助 (Remote Guided Repair)

1. 专家在云端控制台选中某台在线设备，发起远程诊断会话。
2. 专家通过云端下发诊断脚本，获取设备状态。
3. 专家编写修复方案（如修改 Nginx 配置）并下发。
4. 终端桌面弹出完整修复计划："IT 专家建议修改 Nginx 配置，共 2 步，风险等级 L1"。
5. 员工审批后，方案按序执行，每步结果实时回传给专家。

### 场景三：自然语言定义关注点 — 个性化主动巡检

1. 开发者小王刚入职，打开 Desktop 对话框输入："帮我盯着 MySQL 和 Redis 能不能连上，C 盘别超过 80%，还有公司 VPN 别断"。
2. Agent 调用 LLM 将自然语言拆解为 5 项检测：MySQL 连通性、Redis 连通性、C 盘使用率、VPN 网卡状态、MySQL 进程资源（LLM 主动补充建议）。
3. 小王在确认卡片中看到每项的检测方式、阈值和频率，把磁盘阈值从 80% 调整为 85%，删除了 MySQL 进程资源监控，点击"确认"。
4. 两小时后，Agent 巡检发现 Redis 连接失败（连续 2 次），Desktop 弹出通知："⚠️ Redis 服务无响应"。
5. 小王点击通知，看到自动生成的修复计划（检查进程 → 重启 Redis → 验证连接），一键审批后问题自动修复。
6. 第二天小王说"再帮我加一个 Nginx 的监控"，LLM 增量拆解并追加到巡检列表。

### 场景四：主动发现 — 内置规则防患未然

1. Agent Core 在例行扫描中发现：磁盘 C 盘使用率达到 92%，触发内置规则 `PERF-001`。
2. 健康评分从 85 降至 72，Desktop 弹出系统通知："发现磁盘空间不足，建议清理"。
3. 用户点击通知，看到自动生成的修复建议：清理临时文件、清理 npm/pip 缓存。
4. 用户审批后，Agent 自动执行清理，磁盘使用率降至 78%，健康评分恢复。

### 场景五：企业级批量合规治理 (Enterprise Governance)

1. 管理员在云端配置治理规则包："禁止 SSH 密码登录"、"防火墙必须启用"、"TLS 证书不得在 30 天内过期"。
2. 规则包下发到所有终端 Agent，Agent 在本地执行合规扫描。
3. 平台仪表盘实时展示：200 台设备中 15 台存在合规问题，按严重度排序。
4. 违规设备自动收到修复建议推送，员工确认后一键修复。
5. 管理员在审计台追踪每台设备的合规修复进度。

## 5. 安全架构

EnvNexus 的安全模型建立在**零信任**原则之上：

### 5.1 默认只读

Agent 注册的 33 个工具中，26 个为只读（L0），仅 7 个具有写入能力。所有写操作必须经过审批。

### 5.2 分层审批

| 风险等级 | 含义 | 审批要求 | 示例 |
|---------|------|---------|------|
| L0 | 只读，无副作用 | 无需审批 | 读取网络配置、查看进程列表 |
| L1 | 低风险写操作 | 计划级审批后自动执行 | 修改配置文件、切换代理 |
| L2 | 中风险，可能影响服务 | 计划级审批 + 执行前确认 | 重启服务、清理缓存、刷新 DNS |
| L3 | 高风险，不可逆 | 计划级审批 + 每步单独审批 | 删除文件、修改注册表 |

### 5.3 本地策略拦截

即使 LLM 建议执行某操作，本地策略引擎会进行二次校验：
- 修复计划中的所有工具必须在注册表中（LLM 不能发明工具）
- 风险等级以注册表为准（LLM 不能降低风险等级）
- 管理员可通过 Policy Profile 配置额外拦截规则

### 5.4 全链路审计

从诊断发起到修复完成，每个环节都生成结构化审计事件：
- 诊断请求（who/when/what）
- AI 推理过程（evidence/findings/confidence）
- 修复计划（steps/risk/rollback）
- 用户审批决策（approve/deny/timestamp）
- 执行结果（success/failure/rollback）
- 验证结果（pass/fail）

### 5.5 配置加密

- **Windows**: DPAPI 机器绑定加密
- **macOS/Linux**: AES-256-GCM，密钥从 OS 级密钥链派生
- 配置文件拷贝到其他机器后无法解密

## 6. 商业化与交付模式

EnvNexus 提供灵活的商业交付模式，以适应不同规模企业的合规要求：

1.  **SaaS 托管版 (Hosted)**：
    *   开箱即用，按设备并发连接数或活跃设备数（MAU）按月订阅计费。
    *   适合中小企业和初创团队。
2.  **混合云版 (Hybrid)**：
    *   控制台和信令网关由 EnvNexus 托管。
    *   **模型与数据驻留本地**：客户可配置使用企业内部的私有大模型（通过 Ollama 或兼容 OpenAI API 的私有部署），且敏感审计日志直接写入客户的私有 S3 存储，不经过公有云。
3.  **完全私有化部署 (Private)**：
    *   提供完整的 Kubernetes Helm Chart，客户在内网完全独立部署。
    *   通过 License Key 进行商业授权激活。
    *   适合金融、军工、政府等强监管行业。

## 7. 用户旅程 (User Journey)

1.  **入驻**：企业 IT 管理员注册 EnvNexus 云平台，创建租户。
2.  **配置**：配置 AI 模型接入（支持 6+ Provider）、安全策略和治理规则包。
3.  **分发**：管理员在后台点击"生成下载链接"，将链接发送给全员。
4.  **安装**：员工下载客户端（安装包已在云端动态注入了该企业的身份信息），双击安装后**自动完成绑定**，无需输入任何激活码。
5.  **设定关注点**：员工用自然语言告诉 Agent 需要关注什么（如"帮我盯着 MySQL 和磁盘"），Agent 拆解为检测项，员工确认后开始持续巡检。
6.  **日常使用**：客户端静默运行，持续巡检关注项和内置规则。遇到问题时唤起界面，描述问题或粘贴截图，审批修复计划，一键解决。
7.  **主动治理**：Agent 发现异常时通过系统通知提醒用户，并自动生成修复建议。用户也可随时用自然语言追加新的关注点。
8.  **自动更新**：管理员在平台上传新版本 Agent 后，终端客户端自动检测到更新、下载并提示用户一键安装。
9.  **审计与复盘**：IT 部门通过云端控制台查看全局健康态势、导出脱敏审计报表，用于合规检查。

## 9. 与竞品的差异化

| 维度 | 传统远控 (TeamViewer) | IT 工单系统 (ServiceNow) | EnvNexus |
|------|---------------------|------------------------|----------|
| 诊断方式 | 人工远程操作 | 人工排查 + 知识库搜索 | AI 自动诊断 + 多模态输入 |
| 修复方式 | 远程直接操作 | 人工执行 + 工单流转 | 计划式修复 + 自动执行 |
| 安全模型 | 全权限远控 | 依赖人员权限管理 | 零信任 + 分层审批 + 策略拦截 |
| 主动性 | 完全被动 | 完全被动 | 主动发现 + 预警 + 自动建议 |
| 审计能力 | 录屏（非结构化） | 工单记录（粗粒度） | 全链路结构化审计（步骤级） |
| 回滚能力 | 无 | 依赖人工 | 自动状态快照 + 逆序回滚 |
| 离线能力 | 无 | 无 | 本地 LLM + 内置规则 + 离线诊断 |
| 用户体验 | 被动等待 IT 接管 | 提工单等待响应 | 自助诊断 + 一键审批修复 |
