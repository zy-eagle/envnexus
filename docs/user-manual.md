## 第一部分：Web 控制台操作指南（管理员）

控制台是管理员的统一管理界面。所有页面采用**左侧菜单树 + 右侧内容区**的布局。

左侧菜单分为两个可折叠的分组：

- **租户管理**：所有租户、设备管理、会话管理、审计事件、下载包管理、模型配置、策略配置、Agent 配置、环境治理
- **平台**：系统设置、退出登录

---

### 1. 登录

1. 打开浏览器，访问控制台地址（例如 `http://你的服务器IP:3000`）。
2. 输入**邮箱地址**和**密码**。
3. 点击**登录**。

> 默认管理员账号：`admin@envnexus.io` / `admin123`（首次使用后请及时修改）。

---

### 2. 仪表盘总览

**菜单路径**：仪表盘总览

登录后默认进入此页面，展示三张统计卡片：

| 卡片 | 说明 |
|------|------|
| **设备总数** | 当前租户下已注册的设备数量 |
| **活跃会话** | 当前正在进行中的诊断会话数量 |
| **系统状态** | 平台健康状态（绿色圆点 = 健康） |

下方还有一个**最近活动**面板（当前为占位，后续版本将展示实时动态）。

---

### 3. 租户管理

**菜单路径**：租户管理 → 所有租户

租户是 EnvNexus 的顶层隔离单元。每个企业客户对应一个租户，租户之间的数据完全隔离。

#### 查看租户列表

页面以表格形式展示所有租户：

| 列名 | 说明 |
|------|------|
| 名称 | 租户的显示名称 |
| 标识符 | 租户的唯一标识（slug），用于 URL 和 API |
| 状态 | `active`（活跃）/ `suspended`（已暂停）/ `archived`（已归档） |
| 创建时间 | 租户创建日期 |
| 操作 | 编辑、删除、管理 |

#### 创建租户

1. 点击右上角**创建租户**按钮。
2. 填写：
   - **租户名称**：例如 `某某公司`
   - **租户标识 (Slug)**：例如 `acme-corp`（仅允许小写字母、数字和连字符）
3. 点击**创建**。

#### 编辑租户

1. 点击对应行的**编辑**。
2. 可修改**租户名称**和**状态**（活跃 / 已暂停 / 已归档）。
3. 点击**保存**。

#### 进入租户管理

点击对应行的**管理**，将跳转到该租户的设备管理页面，左侧菜单中的所有子页面（设备、会话、配置等）都将在该租户的上下文中操作。

---

### 4. 模型配置

**菜单路径**：租户管理 → 模型配置

模型配置决定了 Agent 使用哪个 AI 模型来进行诊断分析。

#### 查看模型列表

| 列名 | 说明 |
|------|------|
| 名称 | 配置的显示名称 |
| 提供商 | AI 服务提供商 |
| 模型名称 | 具体使用的模型 |
| 状态 | `active` |
| 操作 | 编辑、删除 |

#### 创建模型配置

1. 点击**添加模型**。
2. 填写以下字段：

| 字段 | 说明 |
|------|------|
| **名称** | 给配置起一个易于识别的名字，例如 `生产环境-GPT4` |
| **提供商** | 下拉选择：`OpenAI`、`Anthropic`、`DeepSeek`、`Local`（本地模型） |
| **基础 URL** | API 地址。选择 OpenAI 时默认为 `https://api.openai.com/v1`；如使用代理或私有部署需手动修改 |
| **模型名称** | 下拉选择，选项随提供商变化（见下表） |
| **API Key** | 对应提供商的 API 密钥（密码输入框，不会明文显示） |

3. 点击**创建**。

**各提供商可选模型：**

| 提供商 | 可选模型 |
|--------|---------|
| OpenAI | `gpt-4o`、`gpt-4-turbo`、`gpt-4`、`gpt-3.5-turbo` |
| Anthropic | `claude-3-5-sonnet`、`claude-3-opus`、`claude-3-sonnet`、`claude-3-haiku` |
| DeepSeek | `deepseek-chat`、`deepseek-reasoner` |
| Local | `llama3`、`qwen2`、`mistral` |

> **提示**：切换提供商时，模型名称会自动重置为该提供商的默认模型。

### 5. 策略配置

**菜单路径**：租户管理 → 策略配置

策略配置决定了 Agent 在终端设备上的行为权限边界。

#### 查看策略列表

| 列名 | 说明 |
|------|------|
| 名称 | 策略的显示名称 |
| 模式 | `只读`（蓝色标签）或 `完整访问`（琥珀色标签） |
| 写入工具 | `已启用`（绿色标签）或 `已禁用`（红色标签） |
| 状态 | `活跃` 或 `已归档` |
| 操作 | 编辑、删除 |

#### 创建策略配置

1. 点击**添加策略**。
2. 填写以下内容：

**名称**：给策略起一个名字，例如 `默认安全策略`。

**默认模式**（二选一，点击卡片选择）：

| 模式 | 说明 |
|------|------|
| **只读模式** | Agent 仅能运行只读诊断工具（如查看网络配置、检查进程列表），不允许任何系统修改。**推荐用于大多数场景。** |
| **完整模式** | Agent 可运行所有工具。写入操作仍需用户审批后才能执行。 |

**允许写入工具**（开关控件）：

| 状态 | 说明 |
|------|------|
| **开启**（默认） | AI 可以提出修复建议（如刷新 DNS、重启服务），但每个修复动作在执行前都必须经过用户明确审批 |
| **关闭** | 完全禁用写入类工具，AI 即使发现问题也只能给出建议，无法发起修复请求 |

3. 点击**创建**。

#### 编辑策略

编辑时会额外显示**状态**下拉框，可将策略设置为 `活跃` 或 `已归档`。

### 6. Agent 配置

**菜单路径**：租户管理 → Agent 配置

Agent 配置将**模型配置**和**策略配置**组合在一起，形成一个完整的客户端运行方案。生成安装包时需要指定 Agent 配置。

#### 查看 Agent 配置列表

| 列名 | 说明 |
|------|------|
| 名称 | 配置名称 |
| 更新通道 | `Stable`（稳定版）/ `Beta`（测试版）/ `Dev`（开发版） |
| 状态 | `active` |
| 操作 | 编辑、删除 |

#### 创建 Agent 配置

1. 点击**添加 Agent 配置**。
2. 填写以下字段：

| 字段 | 说明 |
|------|------|
| **名称** | 例如 `标准客户端配置` |
| **模型配置 ID** | 填入之前创建的模型配置的 ID（可从模型配置列表页复制） |
| **策略配置 ID** | 填入之前创建的策略配置的 ID |
| **更新通道** | 选择 `Stable`（推荐）、`Beta` 或 `Dev` |
| **能力清单 (JSON)** | 默认值通常无需修改：`{"local_ui": true, "repair_enabled": true}` |

3. 点击**创建**。

### 7. 下载包管理（生成客户端安装包）

**菜单路径**：租户管理 → 下载包管理

这是将客户端分发给员工的关键步骤。系统会自动交叉编译 Agent 二进制并上传到对象存储，创建安装包时后台会异步构建定制化的安装包（注入租户配置和激活码），构建完成后即可下载。

#### 查看已生成的安装包

| 列名 | 说明 |
|------|------|
| 包名称 | 安装包文件名，格式为 `envnexus-agent-{平台}-{架构}-{版本}` |
| Agent 配置 | 关联的 Agent 配置名称 |
| 平台 | 操作系统/架构（如 `windows/amd64`） |
| 版本 | 安装包版本号 |
| 激活模式 | `自动激活`（蓝色）/ `设备码绑定`（紫色）/ `自动 + 手动`（靛蓝色） |
| 已绑定 | 已绑定设备数 / 最大设备数 |
| 构建状态 | 见下方说明 |
| 操作 | 绑定设备、审计日志、下载、删除 |

**构建状态说明：**

| 状态 | 颜色 | 说明 |
|------|------|------|
| **等待中** | 黄色 | 安装包已创建，等待后台构建任务处理 |
| **构建中** | 蓝色 | 后台正在编译和打包安装包 |
| **已就绪** | 绿色 | 构建完成，可以下载 |
| **失败** | 红色 | 构建失败，请检查后台日志 |

> **自动刷新**：当列表中存在"等待中"或"构建中"状态的安装包时，页面会每 5 秒自动刷新一次，构建完成后自动停止刷新。

#### 生成安装包

1. 点击右上角**生成链接**按钮。
2. 在弹窗中填写：

| 字段 | 说明 |
|------|------|
| **Agent 配置** | 下拉选择之前创建的 Agent 配置 |
| **平台** | 选择 `Linux`、`Windows` 或 `macOS` |
| **架构** | 选择 `amd64` 或 `arm64` |
| **版本** | 默认 `0.1.0` |
| **激活模式** | 勾选一种或两种（见下方说明） |
| **最大设备数** | 该安装包最多可绑定的设备数量 |

3. 点击**生成链接**提交。

**激活模式说明：**

| 模式 | 说明 |
|------|------|
| **自动激活** | 员工安装后，Agent 使用内置激活码自动完成激活，无需任何手动操作 |
| **设备码绑定** | 管理员在控制台手动输入设备码绑定设备后，Agent 才能激活 |
| **同时勾选** | 两种方式均可使用，灵活性最高 |

#### 激活码

选择了"自动激活"模式后，创建成功时会弹出激活码窗口：

- 激活码**仅显示一次**，请务必妥善保存
- 点击**复制**按钮可将激活码复制到剪贴板
- 激活码会被内置到安装包中，员工安装后自动激活

> ⚠️ **重要**：关闭激活码窗口后将无法再次查看，请确保已保存。

#### 下载安装包

- 构建状态为**已就绪**时，操作列会显示**下载**按钮
- 点击**下载**，系统会生成一个 15 分钟有效的临时下载链接并自动开始下载
- 构建中的安装包会显示灰色的"构建中..."提示，无法下载

#### 绑定设备

对于"设备码绑定"模式的安装包：

1. 点击操作列的**绑定设备**
2. 在弹窗中输入设备码（格式：`ENX-XXXX-XXXX-XXXX`）
3. 点击**绑定**

已绑定的设备会显示在列表中，包含设备码、绑定者、绑定时间和最后心跳时间。可点击**解绑**移除设备。

#### 修改最大设备数

在绑定设备弹窗中，可点击**修改最大设备数**调整该安装包允许绑定的最大设备数量。

#### 审计日志

点击操作列的**审计日志**，可查看该安装包相关的所有操作记录（创建、绑定、解绑、激活等）。

#### 删除安装包

点击操作列的**删除**按钮，确认后将永久删除该安装包记录。此操作不可撤销。

> **分发给员工**：安装包构建完成后，下载安装包文件，通过企业微信、钉钉、邮件等方式发送给员工。员工安装后，客户端会自动绑定到你的企业租户下。

### 8. 设备管理

**菜单路径**：租户管理 → 设备管理

当员工安装并运行客户端后，设备会自动出现在此列表中。

#### 查看设备列表

页面顶部显示**在线**（绿色脉冲圆点）和**离线**（灰色圆点）设备数量统计。

| 列名 | 说明 |
|------|------|
| （状态指示灯） | 绿色脉冲 = 在线（5 分钟内有心跳），灰色 = 离线 |
| 设备名称 | 设备的显示名称 |
| 主机名 | 操作系统主机名 |
| 平台 | 操作系统/架构（如 `windows/amd64`） |
| Agent 版本 | 客户端版本号 |
| 状态 | `active`（活跃）/ `pending_activation`（待激活）/ `quarantined`（隔离）/ `revoked`（已撤销） |
| 最后在线 | 最后一次心跳时间 |
| 操作 | 编辑、撤销 |

> **自动刷新**：设备列表每 **30 秒**自动刷新一次，无需手动操作。

#### 编辑设备

可修改**设备名称**和**状态**。

#### 撤销设备

点击**撤销**后确认，该设备将被移除，其客户端将无法再连接到平台。

#### 添加设备

点击**添加设备**会弹出提示：要添加新设备，请在目标机器上下载并安装 EnvNexus Agent，前往**下载包管理**页面生成注册链接。

### 9. 会话管理

**菜单路径**：租户管理 → 会话管理

每次诊断交互都会产生一个会话记录。

#### 查看会话列表

| 列名 | 说明 |
|------|------|
| 设备 ID | 发起诊断的设备 |
| 传输方式 | 通信协议（如 WebSocket） |
| 状态 | `created` / `attached` / `diagnosing` / `executing` / `awaiting_approval` / `completed` / `aborted` / `expired` |
| 发起方 | 会话发起者类型 |
| 开始时间 | 会话创建时间 |
| 操作 | 查看、终止 |

#### 查看会话详情

点击**查看**进入会话详情页，包含：

- **基本信息卡片**：会话 ID、设备 ID、传输方式、发起方、开始时间、持续时间
- **事件时间线**：按时间顺序展示该会话中的所有事件（诊断请求、工具调用、审批决策等），点击每个事件可展开查看完整的 JSON 载荷

#### 终止会话

对于仍在进行中的会话，可点击**终止**按钮强制结束。

### 10. 审计事件

**菜单路径**：租户管理 → 审计事件

所有设备上的每一次诊断、工具调用、审批决策都会生成审计事件，记录在此。

#### 查看审计事件

| 列名 | 说明 |
|------|------|
| 事件类型 | 如 `session.created`、`tool.executed`、`approval.approved` 等 |
| 设备 ID | 关联的设备 |
| 会话 ID | 关联的会话 |
| 创建时间 | 事件发生时间 |
| 操作 | 查看载荷 |

点击**载荷**可展开查看该事件的完整 JSON 数据。

#### 筛选审计事件

页面顶部提供四个筛选条件：

| 筛选项 | 说明 |
|--------|------|
| 按会话 ID 筛选 | 输入会话 ID，只显示该会话的事件 |
| 按设备 ID 筛选 | 输入设备 ID，只显示该设备的事件 |
| 按事件类型筛选 | 输入事件类型关键字 |
| 包含已归档 | 勾选后显示已归档的历史事件 |

填写后点击 **Filter** 按钮执行筛选。

## 第二部分：桌面客户端操作指南（终端用户）

桌面客户端是员工日常使用的工具。安装后它会常驻在系统托盘中，需要时打开即可。

客户端界面采用**左侧导航栏 + 右侧内容区**的布局。左侧导航栏显示：
- **EnvNexus** 品牌名
- **连接状态**指示灯（绿色 = 在线 / 灰色 = 离线 / 橙色 = 连接中）
- 五个导航项：仪表盘、诊断对话、待审批、历史会话、设置

---

### 1. 安装与启动

1. 从管理员提供的下载链接下载安装包。
2. 双击安装，安装完成后客户端会自动启动。
3. 客户端启动后会：
   - 在系统托盘（任务栏右下角）显示图标
   - 自动连接到企业的 EnvNexus 平台
   - 自动绑定到企业租户（无需输入激活码）

---

### 2. 仪表盘

**导航**：点击左侧**仪表盘**

显示四张状态卡片：

| 卡片 | 说明 |
|------|------|
| **连接状态** | 当前与平台的连接状态，以及设备 ID |
| **运行工具数** | 可用的诊断工具数量（只读/写入分类） |
| **待审批** | 当前等待你审批的修复操作数量 |
| **治理基线** | 环境治理基线状态和漂移检测信息 |

下方还有两张详情卡片：
- **Agent Core 状态**：显示平台连接、LLM 提供商、当前会话、待审批等详细信息
- **最新诊断发现**：显示最近的诊断结果列表

> 如果客户端处于离线状态，页面顶部会显示黄色警告条："agent-core 不可达，离线模式"。

### 3. 诊断对话（核心功能）

**导航**：点击左侧**诊断对话**

这是你与 AI 助手交互的核心界面，类似于聊天窗口。

#### 发起诊断

1. 在底部输入框中，用自然语言描述你遇到的问题。
   - 例如：`我的网络连不上了`、`Docker 启动失败`、`帮我检查系统状态`
2. 点击**发送**按钮（或按 **Enter** 键）。
3. AI 助手会显示"分析中..."，然后返回诊断结果。

#### 查看诊断结果

AI 的回复会以对话气泡形式展示。如果 AI 发现了具体问题，页面下方会出现**诊断发现**区域，以结构化卡片展示：
- **问题标题**
- **详细描述**
- **建议的修复方案**（如果有）

#### 多轮对话

你可以继续追问或提供更多信息，AI 会结合上下文给出更精准的诊断。

> **安全提示**：在这个阶段，AI 只会执行**只读**操作（如查看网络配置、检查进程），不会修改你的任何系统配置。

### 4. 待审批操作

**导航**：点击左侧**待审批**

当 AI 认为需要执行修复操作时（如刷新 DNS、重启服务），它不会直接执行，而是生成一个审批请求，等待你的授权。

#### 查看待审批列表

每个审批请求以卡片形式展示，包含：

| 信息 | 说明 |
|------|------|
| **工具名称** | 将要执行的操作（如 `dns.reset`、`service.restart`） |
| **请求 ID** | 唯一标识 |
| **风险等级** | 彩色标签：L0（最低）→ L3（最高） |
| **状态** | 当前审批状态 |
| **原因** | AI 建议执行此操作的原因 |
| **参数详情** | JSON 格式的具体执行参数 |

#### 审批操作

每个请求下方有两个按钮：

- **✓ 批准**：授权 Agent 执行该修复操作
- **✗ 拒绝**：拒绝执行，AI 不会对你的系统做任何修改

点击**刷新**按钮可手动刷新审批列表。

> **重要**：只有你点击了**批准**，修复操作才会真正执行。这是 EnvNexus 的核心安全机制。

## 第四部分：部署指南（运维人员）

本部分面向负责部署和维护 EnvNexus 平台的运维人员。

---

### 1. 环境要求

| 组件 | 最低要求 |
|------|---------|
| **操作系统** | Linux（推荐 Ubuntu 20.04+）、macOS、Windows（需 WSL2） |
| **Docker** | Docker Engine 20.10+ 且 Docker Compose V2 |
| **内存** | 4 GB（推荐 8 GB） |
| **磁盘** | 10 GB 可用空间 |
| **网络** | 服务器需要对外开放端口 3000、8080、8081、9000 |

---

### 2. 一键部署

EnvNexus 提供一键部署脚本，自动完成环境配置、服务构建和启动：

```bash
git clone https://github.com/zy-eagle/envnexus.git
cd envnexus
./deploy.sh start
```

脚本会自动完成以下工作：

1. **检测环境**：确认 Docker 和 Docker Compose V2 已安装
2. **生成配置**：自动检测宿主机 IP，生成 `deploy/docker/.env`（含自动生成的密钥和密码）
3. **构建服务**：编译所有后端服务和前端
4. **编译 Agent**：交叉编译 6 个平台/架构组合的 Agent 二进制（linux/windows/darwin × amd64/arm64）
5. **上传基础包**：将编译好的 Agent 二进制上传到 MinIO 对象存储
6. **启动所有服务**：MySQL、Redis、MinIO、Platform API、Session Gateway、Job Runner、Console Web

部署完成后会显示访问地址：

```
========================================
       EnvNexus is running! 🎉
========================================

  Console Web:      http://192.168.1.100:3000
  Platform API:     http://192.168.1.100:8080
  Session Gateway:  ws://192.168.1.100:8081
```

---

### 3. 部署管理命令

所有管理操作通过 `deploy.sh` 脚本执行：

| 命令 | 说明 |
|------|------|
| `./deploy.sh start` | 一键部署并启动所有服务（首次使用） |
| `./deploy.sh stop` | 停止所有服务（数据保留在 volumes 中） |
| `./deploy.sh restart` | 重启所有服务（不重新构建） |
| `./deploy.sh status` | 查看所有服务运行状态 |
| `./deploy.sh logs` | 查看所有服务日志（实时跟踪） |
| `./deploy.sh logs <服务名>` | 查看指定服务日志，如 `./deploy.sh logs platform-api` |
| `./deploy.sh web` | 仅重新构建并部署前端（Console Web） |
| `./deploy.sh api` | 仅重新构建并部署后端服务（Platform API、Session Gateway、Job Runner） |
| `./deploy.sh agents` | 强制重新编译 Agent 二进制并上传到 MinIO |
| `./deploy.sh reset` | ⚠️ 删除所有数据和配置，恢复到全新状态 |

**常用场景：**

- **更新前端代码后**：`git pull && ./deploy.sh web`
- **更新后端代码后**：`git pull && ./deploy.sh api`
- **更新 Agent 代码后**：`git pull && ./deploy.sh agents`
- **全量更新**：`git pull && ./deploy.sh start`

> **关于 Agent 构建缓存**：`./deploy.sh start` 会自动构建 Agent，但使用 Docker 层缓存——如果 `apps/agent-core/` 源码没有变化，构建会在几秒内完成。上传时也会检测 MinIO 中是否已有相同大小的文件，相同则跳过。只有 Agent 源码实际变更时才会触发完整编译。

### 4. 环境变量配置

部署时自动生成的 `deploy/docker/.env` 包含所有配置项。以下是关键配置说明：

#### 数据库

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `ENX_DATABASE_DSN` | MySQL 连接字符串 | 自动生成 |
| `MYSQL_ROOT_PASSWORD` | MySQL root 密码 | 自动生成 |

#### 对象存储（MinIO）

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `ENX_OBJECT_STORAGE_ENDPOINT` | MinIO 内部地址（Docker 网络） | `minio:9000` |
| `ENX_OBJECT_STORAGE_PUBLIC_ENDPOINT` | MinIO 外部地址（浏览器访问） | `{宿主机IP}:9000` |
| `ENX_OBJECT_STORAGE_BUCKET` | 存储桶名称 | `envnexus` |
| `MINIO_ROOT_USER` | MinIO 访问密钥 | `minioadmin` |
| `MINIO_ROOT_PASSWORD` | MinIO 密钥 | `minioadmin` |

> **重要**：`ENX_OBJECT_STORAGE_PUBLIC_ENDPOINT` 决定了安装包下载链接中的地址。如果员工无法下载安装包，请检查此配置是否指向了员工网络可达的地址。

#### 安全密钥

| 变量 | 说明 |

| `ENX_JWT_SECRET` | JWT 签名密钥（自动生成） |
| `ENX_DEVICE_TOKEN_SECRET` | 设备令牌密钥（自动生成） |

#### 服务端口

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `ENX_PLATFORM_API_PORT` | Platform API 端口 | `8080` |
| `ENX_SESSION_GATEWAY_PORT` | Session Gateway 端口 | `8081` |
| `ENX_CONSOLE_PORT` | Console Web 端口 | `3000` |

> **提示**：`.env` 文件在首次部署时自动生成，后续部署不会覆盖。如需重新生成，先执行 `./deploy.sh reset`（会清除所有数据）。

### 5. Agent 基础包管理

EnvNexus 的安装包生成流程分为两步：

1. **基础包**：预编译的 Agent 二进制文件，存储在 MinIO 的 `base-packages/` 路径下
2. **定制包**：Job Runner 在基础包上注入租户配置（平台地址、激活码等），生成最终安装包

#### 基础包列表

部署时自动编译并上传以下 6 个基础包：

| 文件名 | 平台 | 架构 |
|--------|------|------|
| `enx-agent-linux-amd64` | Linux | x86_64 |
| `enx-agent-linux-arm64` | Linux | ARM64 |
| `enx-agent-windows-amd64.exe` | Windows | x86_64 |
| `enx-agent-windows-arm64.exe` | Windows | ARM64 |
| `enx-agent-darwin-amd64` | macOS | Intel |
| `enx-agent-darwin-arm64` | macOS | Apple Silicon |

#### 手动管理基础包

如果需要在本地开发环境中手动编译和上传基础包：

```bash
# 方式一：使用 Makefile（仅编译，不上传）
make build-agents
# 编译产物在 bin/agents/ 目录下

# 方式二：使用脚本（编译 + 上传）
ENX_OBJECT_STORAGE_ENDPOINT=localhost:9000 ./scripts/build-upload-agents.sh

# 方式三：仅上传已编译的二进制
ENX_OBJECT_STORAGE_ENDPOINT=localhost:9000 ./scripts/build-upload-agents.sh upload
```

#### 排查构建问题

如果安装包一直停留在"等待中"或"构建中"状态：

1. 检查 Job Runner 是否正常运行：`./deploy.sh status`
2. 查看 Job Runner 日志：`./deploy.sh logs job-runner`
3. 确认基础包已上传到 MinIO：访问 MinIO Console（`http://服务器IP:9001`），检查 `envnexus/base-packages/` 路径
4. 如果基础包缺失，执行：`./deploy.sh agents`

### B. 常见问题

**Q：员工安装客户端后，设备没有出现在控制台的设备列表中？**  
A：请检查：(1) 客户端是否成功启动（托盘图标是否显示）；(2) 连接状态是否为"在线"（绿色）；(3) 平台地址是否配置正确（客户端设置页面）。

**Q：AI 诊断时提示"离线模式"？**  
A：说明客户端无法连接到平台服务器。在离线模式下，仍可使用基础的只读诊断工具。请检查网络连接和平台地址配置。

**Q：审批请求中的风险等级是什么意思？**  
A：L0 = 无风险（只读）；L1 = 低风险（如刷新 DNS、清理缓存）；L2 = 中风险（如重启服务、重载容器）；L3 = 高风险。等级越高，操作对系统的影响越大，请谨慎审批。

**Q：关闭客户端窗口后，诊断还在进行吗？**  
A：是的。关闭窗口只是隐藏界面，客户端仍在系统托盘中运行。Agent Core 会继续执行已批准的操作。要完全退出，请右键托盘图标选择"退出"。

**Q：管理员能看到我的诊断记录吗？**  
A：是的。为了企业 IT 审计与安全合规，所有诊断对话、工具调用和审批决策都会被记录并上传到管理后台的审计事件中。

**Q：安装包一直显示"等待中"或"构建中"，无法下载？**  
A：请检查：(1) Job Runner 服务是否正常运行（`./deploy.sh status`）；(2) MinIO 中是否有基础包（访问 `http://服务器IP:9001` 查看 `base-packages/` 目录）；(3) 如果基础包缺失，执行 `./deploy.sh agents` 重新编译上传。

**Q：点击下载后浏览器无法访问下载地址？**  
A：下载链接使用 MinIO 的预签名 URL。请检查 `.env` 中的 `ENX_OBJECT_STORAGE_PUBLIC_ENDPOINT` 是否配置为员工网络可达的地址（如 `192.168.1.100:9000`），而非 Docker 内部地址（`minio:9000`）。修改后执行 `./deploy.sh api` 重启后端。

**Q：激活码丢失了怎么办？**  
A：激活码仅在创建安装包时显示一次，无法再次查看。如果丢失，需要删除该安装包并重新创建一个新的。

**Q：如何修改默认管理员密码？**  
A：当前版本暂不支持在界面上修改密码。请直接修改数据库中 `users` 表的密码字段（需使用 bcrypt 加密）。

**Q：部署后如何更新到新版本？**  
A：执行 `git pull && ./deploy.sh start` 即可。Docker 会自动检测代码变化，仅重新构建有变更的服务。数据保存在 Docker volumes 中，不会丢失。
