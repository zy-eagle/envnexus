> **文档版本**：v1.2
> **日期**：2026-03-31
> **文档状态**：评审中（评审整改见 §12；**金融/政企合规**见 §13）
> **关联设计文档**：`docs/omnidev-state/main/04-design.md` (v2.3)

## 1. 概述

### 1.1 业务背景

EnvNexus 平台当前支持运维人员通过控制台发起远程诊断会话，但缺少**直接向设备下发执行命令**的能力。在实际运维场景中，运维人员经常需要在一台或多台设备上执行特定命令（如重启服务、清理缓存、更新配置），且这些操作需要经过**上级审批**后才能执行，以确保组织管控和操作安全。

### 1.2 核心场景

```
运维人员（王五）需要在 3 台生产服务器上重启 nginx。

1. 王五在控制台创建命令任务，选择目标设备，填写命令
2. 系统根据审批策略，将审批请求发送给运维主管（张三）
3. 张三在控制台"待审批"页面 或 飞书审批卡片中点击"通过"
4. 命令自动下发到 3 台设备执行
5. 执行结果回传，通知王五和张三
```

### 1.3 与现有"本地对话审批"的区别

本模块是**全新的组织级审批系统**，与 agent-desktop 上已有的"本地对话确认"是两套独立机制：

| 维度 | 本地对话确认（已有） | 远程命令审批（本模块） |
|------|---------------------|----------------------|
| **谁触发** | 设备用户自己在 Desktop 对话 | 运维人员通过控制台/IM Bot |
| **谁审批** | 设备用户自己（确认高危操作） | 运维的上级/指定审批人 |
| **审批目的** | 防止 AI 误操作（安全确认） | 组织管控（权限审批） |
| **审批渠道** | Desktop 聊天流内联确认弹窗 | 控制台"待审批"页面 + IM 审批卡片 |
| **设备范围** | 单台（本机） | 单台或批量多台 |
| **本模块是否改动** | 不改动，完全保持现有逻辑 | **本模块设计重点** |

## 2. 角色与权限

### 2.1 涉及角色

| 角色 | 说明 | 典型用户 |
|------|------|---------|
| **运维人员** | 创建命令任务，选择目标设备，填写命令 | 运维工程师、IT 支持专家 |
| **审批人** | 审核运维人员提交的命令任务，通过或拒绝 | 运维主管、技术总监 |
| **管理员** | 配置审批策略、IM 渠道、管理权限 | 租户管理员、IT 负责人 |

### 2.2 权限定义

| 权限标识 | 说明 | 默认角色 |
|---------|------|---------|
| `command:create` | 创建命令任务 | Operator |
| `command:approve` | 审批命令任务 | 审批人角色（由审批策略指定） |
| `command:emergency` | 使用紧急模式（跳过审批） | SRE / 高级运维 |
| `command:bypass_approval` | 免审批特权（所有命令自动通过） | 超级管理员 |
| `policy:manage` | 配置审批策略 | Tenant Admin |
| `integration:manage` | 配置 IM 渠道凭据 | Tenant Admin |
| `audit:read_sensitive` | 查看未脱敏的执行输出/敏感审计字段（可选，v1.1） | 合规/安全管理员 |
| `command:adhoc_shell` | 在生产等受控环境使用**非模板**自由文本命令（可选） | 经授权的高级运维 |

## 3. 功能模块

### 3.1 命令任务管理

#### 3.1.1 创建命令任务

**入口**：控制台 → 侧边栏 → 命令任务 → [+ 新建任务]

**交互流程**：

1. 运维人员点击"新建任务"，弹出创建表单
2. 填写以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 任务标题 | 文本 | 是 | 简要描述任务目的，如"重启生产环境 nginx" |
| 命令类型 | 单选 | 是 | Shell 命令 / 工具调用 |
| 命令内容 | 文本 | 是 | 具体命令，如 `systemctl restart nginx` |
| 目标设备 | 多选 | 是 | 从设备列表中勾选，或按标签批量选择 |
| 风险等级 | 单选 | 是 | L1 (低) / L2 (中) / L3 (高)，系统可自动建议 |
| 变更单/工单号 | 文本 | 策略决定 | **RFC / 变更申请 / ITIL 工单 / 内部编号**。金融/政企场景下常 **强制绑定**（见 §13.2）；与外部系统对齐时可存 `external_ticket_url`。 |
| 业务系统/应用 | 文本或下拉 | 否 | 便于审计检索与影响面分析；可与 CMDB/应用台账关联。 |
| 目标环境 | 单选 | 否 | **生产 / 预发 / 测试 / 灾备** 等；命中「生产」时触发 **升级审批/禁用自由命令** 等策略（见 §12.3、§13）。 |
| 备注 | 文本 | 否 | 补充说明 |

3. 提交后，系统根据 **服务端有效风险 `effective_risk`**（§12.1）匹配审批策略：
   - 若匹配到策略且 **同时满足** `auto_approve` 全部风控条件（§12.1）→ 命令立即下发执行
   - 否则若策略指定了**审批人/审批规则** → 任务进入「待审批」，通知审批人（控制台 + IM）
   - 若未配置策略 → 默认需要租户管理员审批

4. 运维人员可在任务列表中查看任务状态

**表单 UI 示意**：

```
┌─────────────────────────────────────────┐
│  新建命令任务                              │
│                                          │
│  任务标题: [重启生产环境 nginx        ]     │
│                                          │
│  命令类型: ○ Shell 命令  ○ 工具调用         │
│                                          │
│  命令内容: [systemctl restart nginx   ]     │
│                                          │
│  目标设备: [✓] 设备A  [✓] 设备B  [ ] 设备C  │
│            或: 按标签选择 [生产环境 ▼]       │
│                                          │
│  风险等级: ○ L1 低  ● L2 中  ○ L3 高       │
│                                          │
│  审批人: [张三 (运维主管) ▼] (自动匹配策略)  │
│                                          │
│  备注: [紧急修复，nginx 502 错误     ]      │
│                                          │
│  [取消]                    [提交审批]       │
└─────────────────────────────────────────┘
```

**风险等级自动建议规则**：

| 风险等级 | 命令特征 | 示例 |
|---------|---------|------|
| L1 (低) | 只读诊断命令 | `systeminfo`, `df -h`, `ping`, `cat /etc/os-release` |
| L2 (中) | 服务操作 | `systemctl restart nginx`, 清缓存, 重启进程 |
| L3 (高) | 系统级操作 | 修改配置文件, 安装/卸载软件, `docker compose up -d` |

> **风险等级最终生效规则（评审整改）**：系统根据命令内容与租户策略计算 **服务端有效风险等级 `effective_risk`**（只上调、不下调）。运维人员在表单中看到的等级为 **UI 建议值**；若用户希望将等级 **调低**（例如从 L3 改为 L2），必须走 **降级申请**：说明理由，并由策略规定的审批人（通常与 `command:approve` 同级或更高）在独立流程中批准降级后，任务才能以低等级参与策略匹配。严禁仅靠前端修改等级绕过 L3/L2 审批。

#### 3.1.2 命令任务列表

**入口**：控制台 → 侧边栏 → 命令任务

**功能**：
- 展示当前租户下所有命令任务（运维视角：我创建的任务）
- 支持按状态筛选：全部 / 待审批 / 已通过 / 已拒绝 / 执行中 / 已完成 / 已过期 / 已取消
- 支持按关键词搜索（任务标题、命令内容）
- 每条任务显示：标题、命令、目标设备数、风险等级、状态、创建时间、审批人

**列表项交互**：
- 点击任务 → 进入任务详情页
- "待审批"状态的任务 → 运维人员可点击"取消"撤回

#### 3.1.3 命令任务详情

**入口**：命令任务列表 → 点击某条任务

**展示内容**：

| 区域 | 内容 |

| **执行结果** | 每台设备一行，显示：设备名称、执行状态、输出内容、退出码、耗时 |

**设备执行状态**：

### 3.2 待审批（审批人视角）

#### 3.2.1 待审批列表

**入口**：控制台 → 侧边栏 → 待审批（带角标显示待处理数量）

**可见性**：仅拥有 `command:approve` 权限的用户可见此菜单项。

**数据范围**：仅显示当前登录用户作为审批人的、状态为"待审批"的任务。

**交互流程**：

1. 审批人登录控制台后，侧边栏"待审批"入口显示红色角标（如 `3`）
2. 点击进入待审批列表
3. 每条待审批任务显示：
   - 任务标题、命令内容
   - 申请人（运维人员）
   - 目标设备列表
   - 风险等级（L1 绿色 / L2 黄色 / L3 红色）
   - 提交时间（相对时间，如"2 分钟前"）
   - 剩余审批时间（接近超时时显示倒计时警告）
4. 每条任务提供三个操作按钮：
   - **[✅ 通过]** — 弹出确认弹窗，可填写审批备注
   - **[❌ 拒绝]** — 弹出确认弹窗，可填写拒绝原因
   - **[查看详情 →]** — 跳转到任务详情页

**角标更新机制**：
- 页面加载时请求待审批数量
- 每 30 秒轮询一次更新角标数字
- 未来可升级为 WebSocket 实时推送

#### 3.2.2 审批操作

**通过操作**：

1. 审批人点击 [✅ 通过]
2. 弹出确认弹窗：

```
┌─────────────────────────────────────────┐
│  确认通过                                │
│                                          │
│  您即将通过以下命令任务：                  │
│  「重启生产环境 nginx」                    │
│  命令将下发到 3 台设备执行。               │
│                                          │
│  审批备注: [同意执行，注意观察日志    ]     │
│  (选填)                                  │
│                                          │
│  [取消]                    [确认通过]     │
└─────────────────────────────────────────┘
```

3. 确认后：
   - 任务状态变为"已通过"
   - 命令自动下发到目标设备
   - 通知运维人员：任务已通过
   - 待审批列表移除该任务，角标数字 -1

**拒绝操作**：

1. 审批人点击 [❌ 拒绝]
2. 弹出确认弹窗：

```
┌─────────────────────────────────────────┐
│  确认拒绝                                │
│                                          │
│  您即将拒绝以下命令任务：                  │
│  「重启生产环境 nginx」                    │
│                                          │
│  拒绝原因: [请先确认 nginx 配置无误   ]    │
│  (选填)                                  │
│                                          │
│  [取消]                    [确认拒绝]     │
└─────────────────────────────────────────┘
```

3. 确认后：
   - 任务状态变为"已拒绝"
   - 命令不会执行
   - 通知运维人员：任务已被拒绝（附带拒绝原因）

#### 3.2.3 审批超时

- 每条审批任务有超时时间（由审批策略配置，默认 30 分钟）
- 接近超时时（剩余 5 分钟），列表项显示橙色倒计时警告
- 超时后：
  - 任务状态自动变为"已过期"
  - 通知运维人员：审批已超时
  - 运维人员可重新提交任务

### 3.3 审批策略管理

#### 3.3.1 功能说明

审批策略定义了**什么风险等级的命令需要谁来审批**。这是审批人配置的核心入口。

**入口**：控制台 → 侧边栏 → 审批策略

**权限**：`policy:manage`（管理员）

#### 3.3.2 策略列表

展示当前租户下所有审批策略，每条策略显示：

| 列 | 说明 |
|----|------|
| 策略名称 | 如"中风险运维主管审批" |
| 风险等级 | L1 / L2 / L3 / *（所有） |
| 审批人 | 指定用户名 或 角色名 |
| 自动通过 | 是 / 否 |
| 审批超时 | N 分钟 |
| 优先级 | 数字（越大越优先） |
| 操作 | [编辑] [删除] |

#### 3.3.3 新建/编辑策略

**表单字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 策略名称 | 文本 | 是 | 策略的描述性名称 |
| 风险等级 | 单选 | 是 | L1 (低) / L2 (中) / L3 (高) / * (所有等级) |
| 审批方式 | 单选 | 是 | 指定审批人 / 按角色审批 |
| 审批人 | 下拉选择 | 条件必填 | 选择"指定审批人"时，从用户列表中选择 |
| 审批角色 | 下拉选择 | 条件必填 | 选择"按角色审批"时，从角色列表中选择（该角色下任一用户通过即可） |
| 自动通过 | 开关 | 否 | 开启后跳过审批，命令直接执行（适用于 L1 低风险） |
| 审批超时 | 数字 | 是 | 超时时间（分钟），超时后任务自动过期。默认 30 |
| 优先级 | 数字 | 是 | 同一风险等级有多条策略时，优先级高的先匹配。默认 0 |
| 审批规则 | 单选 | 是 | **单人审批**（默认）/ **双人审批**（会签，两名审批人均需通过）/ **顺序审批**（预审 + 终审，按顺序通过）。L3 默认建议 **双人或顺序**，可由租户关闭（需 `policy:manage` 显式配置并审计）。 |
| 职责分离 | 开关 | 否 | 开启后：**创建人不可审批本人任务**；紧急通道通知对象不得与操作人相同（至少通知另一名管理员）。建议 **L2/L3 默认开启**。 |
| 策略版本快照 | — | 自动 | 任务创建时 **锁定**当时匹配到的策略 ID + 版本号 + 规则参数（见 §12.2），后续策略变更不影响已创建任务。 |

**交互规则**：
- 选择"自动通过"时，审批人/角色字段隐藏（不需要审批人）
- 同一风险等级可配置多条策略，按优先级从高到低匹配，匹配到第一条即停止
- 删除策略时弹出二次确认

#### 3.3.4 策略匹配逻辑

```
运维创建命令任务（风险等级 = L2）
  ↓
查找审批策略（租户 + 风险等级 = L2，按优先级降序）
  ↓
├── 找到策略，auto_approve = true
│     → 自动通过，命令直接下发
├── 找到策略，指定审批人 = 张三
│     → 张三成为审批人，IM 通知 + 控制台待审批可见
├── 找到策略，指定角色 = 运维主管
│     → 若审批规则为**单人**：该角色下**任一**用户通过即可
│     → 若审批规则为**双人**：需**两名不同**的、且具备 `command:approve` 的用户依次/会签通过（详见 §5、`pending_second_approval`）
│     → 若审批规则为**顺序**：按策略配置的顺序逐级通过
└── 未找到 L2 策略 → 查找 * (通配) 策略
      └── 仍未找到 → 默认需要租户管理员审批
```

#### 3.3.5 默认策略建议

系统初始化时，建议为租户预置以下默认策略：

| 策略名称 | 风险等级 | 审批人 | 自动通过 | 超时 |
|---------|---------|--------|---------|------|
| 低风险需审批（默认） | L1 | 租户管理员（或按组织调整） | ❌ 否 | 30 分钟 |
| 中风险需审批 | L2 | 租户管理员 | ❌ 否 | 30 分钟 |
| 高风险需审批 | L3 | 租户管理员 | ❌ 否 | 60 分钟 |

管理员可根据实际组织架构修改审批人。

### 3.4 紧急模式（逃生通道）

#### 3.4.1 业务场景

在生产事故（如服务宕机、安全漏洞）等紧急情况下，运维人员需要**立即执行命令**，不能等待审批流程。系统提供"紧急模式"作为逃生通道。

#### 3.4.2 触发方式

| 触发方式 | 操作 | 所需权限 |
|---------|------|---------|
| 控制台 | 新建任务时勾选"🚨 紧急执行" | `command:emergency` |
| 飞书 Bot | 使用 `/exec!` 命令（感叹号表示紧急） | `command:emergency` |

#### 3.4.3 交互流程

**控制台紧急模式**：

1. 运维人员在新建任务表单中勾选"🚨 紧急执行（跳过审批，直接下发）"
2. 勾选后，表单变化：
   - 出现警告提示："⚠️ 紧急执行将跳过审批流程，命令将立即下发到目标设备。此操作会被完整记录。"
   - 出现"紧急原因"必填字段
   - 提交按钮变为红色 "🚨 立即执行"
3. 填写紧急原因（必填，不能为空）
4. 点击"立即执行"后：
   - 系统验证用户有 `command:emergency` 权限
   - 若无权限 → 提示"您没有紧急执行权限，请联系管理员"
   - 若有权限 → 命令立即下发执行
5. **同时**（并行，不阻塞执行）：
   - 向审批策略中配置的审批人发送**通知卡片**（非审批卡片，仅告知）
   - 写入完整审计日志

```
┌─────────────────────────────────────────┐
│  新建命令任务                              │
│                                          │
│  ... (其他字段) ...                        │
│                                          │
│  ☑ 🚨 紧急执行 (跳过审批，直接下发)         │
│     ⚠️ 紧急执行将跳过审批流程，命令将立即     │
│     下发到目标设备。此操作会被完整记录。       │
│                                          │
│     紧急原因: [生产 nginx 502，需立即重启]   │
│     (必填)                                │
│                                          │
│  [取消]              [🚨 立即执行]          │
└─────────────────────────────────────────┘
```

#### 3.4.4 审批人通知

紧急模式下，审批人会收到**两次通知**：

**第一次 — 命令下发时（立即）**：
- 通知类型：🚨 紧急命令通知（红色标题）
- 内容：操作人、命令、目标设备、风险等级、紧急原因
- 说明："此命令已通过紧急通道直接下发执行"
- 无操作按钮（仅通知，无需审批）

**第二次 — 执行结果出来后**：
- 通知类型：✅/❌ 紧急命令执行完成
- 内容：执行结果、输出内容、耗时
- 标注"紧急通道"标签

#### 3.4.5 安全保障

| 保障措施 | 说明 |
|---------|------|
| 权限控制 | 只有拥有 `command:emergency` 权限的用户才能使用 |
| 强制填写原因 | 紧急原因不能为空 |
| 完整审计存档 | 操作人、命令、设备、原因、执行结果全部记录 |
| 实时通知审批人 | 命令下发时立即通知上级，确保知情 |
| 结果二次通知 | 执行结果出来后再次通知上级 |
| 审计报表 | 管理员可在审计事件页面筛选查看所有紧急下发记录 |
| 权限可收回 | 管理员可随时收回 `command:emergency` 权限 |

#### 3.4.6 其他免审批场景

除紧急模式外，还有两种情况命令会跳过审批：

| 场景 | 条件 | 说明 |
|------|------|------|
| 特权用户 | 用户拥有 `command:bypass_approval` 权限 | 超级管理员/SRE 负责人*，所有命令自动通过 |
| 策略自动通过 | 审批策略配置了 auto_approve = true | 仅当满足 §12.1 **有效风险与 auto_approve 约束**时生效；否则服务端提升等级或拦截 |

\* `bypass_approval` 仍须受 **黑名单 / 法定禁止命令** 约束（若租户启用）。

#### 3.4.7 防滥用与运营要求（评审整改）

| 措施 | 说明 |
|------|------|
| **临时授权（推荐）** | `command:emergency` 可配置 **TTL**（如 4h/24h），到期自动失效；或采用 **工单/理由码** 激活的短期 grant。 |
| **频控与告警** | 按 **用户 / 租户** 限制紧急下发次数与时间窗口（如每用户每日 ≤ N 次）；超限 **阻断或降级为需审批**，并触发告警事件。 |
| **事后补录（推荐）** | 紧急执行完成后 **T 小时内**（如 24h）须由操作人或管理员补交「事故单链接 / 根因简述」；逾期未补录记入合规告警（审计事件，见 §9）。 |
| **L3 + 紧急** | 对 L3 命令，策略可配置：**仍允许紧急**但 **自动提升审计等级** / 或 **禁止自动通过类策略**，以减少「紧急」绕过高危治理。 |

### 3.5 IM 集成管理

IM 集成分为**三层配置**，分别由不同角色在不同位置配置。

#### 3.5.1 第①层：IM 渠道配置（管理员）

**入口**：控制台 → 设置 → 集成管理 Tab

**权限**：`integration:manage`（管理员）

**功能**：配置飞书/企业微信/钉钉等 IM 工具的应用凭据，使系统能够向 IM 发送消息。

**交互流程**：

1. 管理员进入"设置 → 集成管理"
2. 页面展示已配置的 IM 渠道列表，每条显示：
   - IM 类型（飞书/企微/钉钉）+ 图标
   - 名称（如"生产环境飞书机器人"）
   - 连接状态（✅ 已连接 / ⚠️ 未配置 / ❌ 连接失败）
   - App ID（脱敏显示）
   - 操作按钮：[测试连接] [编辑] [删除]
3. 点击"+ 添加渠道"，弹出表单：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 渠道类型 | 下拉 | 是 | 飞书 / 企业微信 / 钉钉 / Slack |
| 名称 | 文本 | 是 | 渠道的描述性名称 |
| App ID | 文本 | 是 | IM 开放平台的应用 ID |
| App Secret | 密码 | 是 | IM 开放平台的应用密钥（加密存储） |
| Webhook URL | URL | 否 | 消息推送地址（部分 IM 需要） |

4. 填写完成后，可点击"测试连接"验证凭据是否正确
5. 保存后，该渠道可用于发送审批通知

**安全说明**：
- App Secret 使用 AES-256-GCM 加密存储
- API 响应中不返回 Secret 明文，仅显示脱敏后的 App ID
- 编辑时 Secret 字段显示为 `••••••••`，不修改则保留原值

#### 3.5.2 第②层：用户 IM 绑定（每个用户）

**入口**：控制台 → 设置 → 通知渠道 Tab

**权限**：任何已登录用户（管理自己的绑定）

**功能**：用户将自己的平台账号与 IM 账号关联，用于接收审批通知和命令执行结果。

**交互流程**：

1. 用户进入"设置 → 通知渠道"
2. 页面展示已绑定的 IM 渠道列表：
   - IM 类型 + 图标
   - IM 用户名
   - 验证状态（✅ 已验证 / ⏳ 待验证）
   - 绑定时间
   - 操作按钮：[解绑]
3. 点击"+ 绑定新渠道"，弹出表单：
   - 选择渠道：下拉列表（仅显示管理员已配置的 IM 渠道）
   - 填写 IM 用户 ID
   - 点击"发送验证码" → 系统通过 IM 向该用户发送验证码
   - 输入验证码 → 确认绑定

**绑定方式汇总**：

| 方式 | 操作 | 适用场景 |
|------|------|---------|
| 控制台手动绑定 | 设置 → 通知渠道 → 绑定新渠道 → 验证码 | 标准流程 |
| IM Bot 自助绑定 | 飞书群发 `/bindme email@co.com` → `/verify 123456` | 最简单，推荐 |
| 管理员代绑 | 管理员在用户管理中为用户配置 | 批量初始化 |

**重要说明**：
- 用户 IM 绑定是**接收通知的前提**。未绑定 IM 的审批人只能在控制台"待审批"页面审批，无法收到 IM 通知。
- 一个用户可以绑定多个 IM 渠道（如同时绑定飞书和企微），系统按优先级选择渠道发送通知。

#### 3.5.3 第③层：审批策略（管理员配置审批人）

**入口**：控制台 → 侧边栏 → 审批策略

详见 §3.3 审批策略管理。

### 3.6 IM 审批卡片

#### 3.6.1 审批请求卡片

当命令任务需要审批时，系统向审批人的 IM 发送交互式审批卡片。

**卡片内容**：

```
┌──────────────────────────────────┐
│  🔧 EnvNexus 命令审批请求          │
│  ─────────────────────────────── │
│  申请人: 王五 (运维工程师)          │
│  目标设备: device-001 (共 1 台)    │
│  命令: systemctl restart nginx    │
│  风险等级: L2 (中) 🟡              │
│  备注: 紧急修复，nginx 502 错误     │
│  提交时间: 2026-03-31 14:30       │
│                                   │
│  [✅ 通过]        [❌ 拒绝]         │
└──────────────────────────────────┘
```

**交互逻辑**：
- 审批人点击 [✅ 通过] → 系统回调 Platform API → 任务通过 → 命令下发
- 审批人点击 [❌ 拒绝] → 系统回调 Platform API → 任务拒绝 → 通知运维
- 审批完成后，卡片更新为已处理状态（按钮变灰，显示审批结果）
- 幂等性：同一任务只能审批一次，重复点击无效；**双人审批**下，同一审批人在同一阶段仅计一次有效操作。

**IM 回调安全要求（评审整改，与 §12.5 一致）**：

- **HTTPS + 签名校验**：Platform API 校验 IM 平台或自建网关的签名；失败则拒绝并记安全审计（与 §7.2 一致）。
- **防重放**：请求携带 **`timestamp` + `nonce`**（或 IM 平台等价机制）；服务端校验时钟窗口（如 ±120s），**同一 `nonce` 在窗口内仅处理一次**（Redis/DB 去重）。
- **幂等键**：业务键建议为 `task_id + action + approver_im_uid + request_id`，重复提交返回同一结果（HTTP 200 + 已处理状态）。
- **动作绑定**：回调载荷必须包含 **任务 ID、动作（approve/deny）、审批人 IM 身份**；服务端将 IM 身份解析为平台用户 ID，且须与 **任务当前阶段的合法审批人集合** 匹配，否则拒绝。

**IM 审批卡片的合规与体验要求（评审整改）**：

- **可解释性（强制）**：卡片必须展示：
  - `effective_risk`（服务端最终有效风险等级）
  - 审批规则（单人 / 双人 / 顺序）与当前阶段（如「第 1/2 人已通过，仍需 1 人」或「预审已通过，等待终审」）
  - 策略快照摘要（`matched_policy_id` / `policy_version` 或简化展示「策略：高风险需审批 v3」）
  - 职责分离/治理拦截结果（若触发限制，应以提示语说明原因）
- **最小可用信息**：目标设备数量、标签/环境提示（如「生产环境」）、命令模板名或命令摘要（自由文本命令需高亮“非模板”标识）。
- **防误触**：通过/拒绝按钮需二次确认（尤其 L3/双人/顺序阶段），并支持填写备注；紧急通道不应出现在审批卡片中（紧急是创建侧决策）。
- **金融/政企 profile（`financial_gov`）**：IM 侧 **不得** 展示完整明文命令与完整执行输出；卡片仅 **摘要 + 带鉴权的跳转链接** 进入控制台查看详情（§13.5）。

#### 3.6.2 执行结果卡片

命令执行完成后，系统向运维人员和审批人发送结果通知。**`financial_gov` 模式下 IM 结果卡片须脱敏**，完整输出仅控制台展示（§13.5）。

**成功**：

```
┌──────────────────────────────────┐
│  ✅ 命令执行完成                    │
│  ─────────────────────────────── │
│  任务: 重启生产环境 nginx          │
│  设备: device-001                 │
│  结果: ✅ 成功 (耗时 1.2s)         │
│  输出: nginx.service restarted    │
└──────────────────────────────────┘
```

**失败**：

```
┌──────────────────────────────────┐
│  ❌ 命令执行失败                    │
│  ─────────────────────────────── │
│  任务: 重启生产环境 nginx          │
│  设备: device-002                 │
│  结果: ❌ 失败 (退出码: 1)          │
│  错误: nginx.service not found    │
└──────────────────────────────────┘
```

#### 3.6.3 控制台审批与 IM 审批的关系

审批人可以在**任一渠道**完成审批，两者等价：

| 审批渠道 | 操作方式 | 说明 |
|---------|---------|------|
| 控制台 | 待审批页面 → 点击通过/拒绝 | 适合在电脑前工作时 |
| IM 卡片 | 飞书/企微/钉钉卡片 → 点击通过/拒绝 | 适合移动端、不在电脑前时 |

- 在任一渠道审批后，另一渠道自动同步状态（IM 卡片按钮变灰，控制台列表移除）
- 不会出现重复审批的情况（后端幂等性保证）

### 3.7 飞书 Bot 命令扩展

在已有的飞书 Bot 基础上，扩展以下命令：

#### 3.7.1 命令下发

| 命令 | 说明 | 示例 |
|------|------|------|
| `/exec <device_id> <command>` | 向单台设备下发命令 | `/exec device-001 systemctl restart nginx` |
| `/exec-batch <tag> <command>` | 向标签匹配的设备批量下发 | `/exec-batch 生产环境 df -h` |
| `/exec! <device_id> <command>` | 紧急下发（跳过审批） | `/exec! device-001 systemctl restart nginx` |
| `/tasks` | 查看我的命令任务列表 | `/tasks` |
| `/task <task_id>` | 查看任务详情和执行结果 | `/task 01HXYZ...` |

#### 3.7.2 自助 IM 绑定

| 命令 | 说明 | 示例 |
|------|------|------|
| `/bindme <email>` | 将当前飞书账号绑定到平台用户 | `/bindme zhangsan@company.com` |
| `/verify <code>` | 输入验证码完成绑定 | `/verify 123456` |

#### 3.7.3 交互示例

**普通命令下发**：

```
运维: /exec device-001 systemctl restart nginx
Bot:  📋 命令任务已创建
      任务 ID: 01HXYZ...
      命令: systemctl restart nginx
      目标: device-001
      风险: L2 (中)
      审批人: 张三 (运维主管)
      状态: ⏳ 等待审批

      审批卡片已发送给张三。
```

**紧急命令下发**：

```
运维: /exec! device-001 systemctl restart nginx
Bot:  ⚠️ 紧急模式: 请输入原因
运维: 生产 502 紧急修复
Bot:  🚨 紧急命令已下发 (跳过审批)
      命令: systemctl restart nginx
      目标: device-001
      原因: 生产 502 紧急修复
      ⚠️ 此操作已通知审批人 张三
```

#### 3.7.4 自然语言交互式创建任务（评审整改）

除 `/exec` 等结构化命令外，IM 端还应支持运维人员用**自然语言**创建命令任务，并在 IM 内完成审批闭环。目标体验：

- 运维人员在 IM 里直接说「帮我重启生产 nginx」「把 A 组机器清理缓存」等，Bot 自动生成任务草稿。
- Bot **默认走审批流程**（任务进入 `pending_approval`），**只有当运维人员显式表达“紧急下发/立即执行”**（或使用 `/exec!`）时才进入紧急通道。
- 大模型负责「理解意图与补全参数」，但**审批策略与风控结论以服务端为准**（`effective_risk`、策略匹配、职责分离等均由后端最终裁决）。

**推荐交互流程**：

1. 运维人员发起自然语言请求（示例：`帮我重启生产环境 nginx，影响 3 台机器`）。
2. Bot 调用大模型完成解析并生成 **任务草稿**：
   - 任务标题、目标设备/标签、候选命令（模板优先）、备注
   - 解析出的 `effective_risk`（仅用于提示，最终仍由服务端计算）
   - 预计审批人/审批规则（根据策略推断，用于预览）
3. Bot 以卡片/消息回显草稿并请求确认：
   - **[提交审批]**（默认）
   - **[修改草稿]**（继续自然语言补充/修正）
   - **[取消]**
   - **[🚨 紧急下发]**（仅当用户明确表示紧急或具备 `command:emergency` 权限时展示；且必须补填原因）
4. 用户点击 **提交审批** 后，系统创建命令任务并进入 `pending_approval`，向审批人发送 **IM 审批卡片**（§3.6）。
5. 审批人可在 IM 卡片点击通过/拒绝；或在控制台待审批页面操作，两者等价（§3.6.3）。

**默认审批原则（与 §12.1 一致）**：
- 默认 **所有自然语言创建的任务** 都需要审批（含 L1），除非用户显式选择紧急通道或管理员显式配置自动通过并满足风控约束。

### 3.8 agent-desktop 变更

#### 3.8.1 移除"待审批"页面

agent-desktop 侧边栏的 **"待审批"** 页面将被移除，因为：
- 远程命令审批在**控制台**进行，不在客户端
- 本地对话的写操作确认是内联在聊天流中的，不需要独立页面

**具体移除内容**：
- 侧边栏"待审批"导航项
- "待审批"页面
- 仪表盘上的"待审批"数字卡片
- 相关的国际化文案

#### 3.8.2 本地对话确认 — 保持不变

Desktop 上的 AI 对话中，用户自己确认高危操作的内联弹窗**完全保持不变**：
- AI 调用写操作工具时 → 聊天流中出现确认卡片
- 用户点击批准/拒绝 → AI 继续/中止
- 这是本地安全确认，与远程命令审批是独立系统

## 4. 通知机制

### 4.1 通知事件总览

| 事件 | 通知对象 | 通知渠道 | 说明 |
|------|---------|---------|------|
| 任务创建（需审批） | 审批人 | IM 审批卡片 + 控制台待审批 | 审批人可在任一渠道操作 |
| 任务被通过 | 运维人员 | IM 通知 | 告知命令即将执行 |
| 任务被拒绝 | 运维人员 | IM 通知 | 附带拒绝原因 |
| 任务超时过期 | 运维人员 | IM 通知 | 提示重新提交 |
| 命令执行完成 | 运维人员 + 审批人 | IM 结果卡片 | 附带执行输出 |
| 紧急下发（命令下发时） | 审批人 | IM 紧急通知卡片 | 仅通知，无需操作 |
| 紧急下发（执行结果） | 审批人 | IM 结果卡片 | 标注"紧急通道" |

### 4.2 通知降级策略

| 场景 | 系统行为 |
|------|---------|
| 审批人已绑定 IM | 通过 IM 发送通知 + 控制台待审批可见 |
| 审批人未绑定 IM | 仅控制台待审批可见，创建任务时提示运维人员"审批人未绑定 IM，请通知其登录控制台审批" |
| 未配置任何 IM 渠道 | 所有审批均在控制台进行，无 IM 通知 |
| IM 发送失败 | 记录错误日志，不影响审批流程（审批人仍可在控制台操作） |

## 5. 状态机

### 5.1 命令任务状态流转

```
                    ┌──────────────┐
                    │   创建任务    │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
              ┌─────┤ pending_approval │◄─── 需要审批
              │     └──────┬───────┘
              │            │
    ┌─────────▼──┐  ┌──────▼───────┐  ┌──────────┐
    │  cancelled  │  │   approved   │  │  denied   │
    │  (运维取消) │  │  (审批通过)   │  │ (审批拒绝) │
    └────────────┘  └──────┬───────┘  └──────────┘
                           │
                    ┌──────▼───────┐
                    │  executing   │◄─── 命令下发中
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
       ┌──────▼──┐  ┌──────▼──┐  ┌──────▼──────┐
       │completed │  │ failed  │  │partial_done │
       │ (全部成功)│  │(全部失败)│  │ (部分成功)   │
       └─────────┘  └─────────┘  └─────────────┘

    特殊路径:
    ─ 自动通过 (auto_approve): 创建 → approved → executing → ...
    ─ 紧急模式 (emergency):    创建 → approved → executing → ...
    ─ 审批超时:                pending_approval → expired
    ─ 双人审批:                pending_approval → pending_second_approval → approved
                                     （第一次通过）    （第二次通过）
```

**说明**：当策略为 **双人审批** 时，第一次通过后进入 `pending_second_approval`，第二位审批人通过后再进入 `approved`；任一步骤可 **拒绝** → `denied`，或 **超时** → `expired`。顺序审批可建模为多阶段 `pending_approval`（阶段字段），或复用 `pending_second_approval` 扩展更多阶段（实现任选其一，须在 `04-design.md` 统一）。

### 5.2 状态流转规则

| 当前状态 | 可转移到 | 触发条件 |
|---------|---------|---------|
| pending_approval | pending_second_approval | 策略为双人/多阶段审批，且**第一阶段**通过 |
| pending_second_approval | approved | 策略要求的所有阶段均已通过（双人场景为第二次通过） |
| pending_second_approval | denied | 任一阶段审批人拒绝 |
| pending_second_approval | expired | 超时 |
| pending_approval | approved | 审批人通过（单人审批规则） |
| pending_approval | denied | 审批人拒绝 |
| pending_approval | cancelled | 运维人员取消 |
| pending_approval | expired | 超时自动过期 |
| approved | executing | 命令开始下发 |
| executing | completed | 所有设备执行成功 |
| executing | failed | 所有设备执行失败 |
| executing | partial_done | 部分设备成功，部分失败/跳过 |

**约束**：状态流转是单向的，不可逆。

## 7. 异常处理与边界条件

### 7.1 设备相关

| 场景 | 系统行为 |
|------|---------|
| 目标设备离线 | 该设备的执行记录标记为 `skipped`，其他在线设备正常执行 |
| 所有目标设备离线 | 任务状态变为 `failed`，通知运维人员 |
| 命令执行超时 | 该设备执行记录标记为 `timeout`（默认超时 60 秒，可在命令中配置） |
| 设备在执行过程中断开 | 执行记录标记为 `failed`，错误信息为"设备连接中断" |

### 7.2 审批相关

| 场景 | 系统行为 |
|------|---------|
| 审批人账号被禁用 | 任务无法审批，运维人员需联系管理员更换审批人或取消任务 |
| 审批策略被删除（任务已创建） | 已创建的任务不受影响，仍按原审批人处理 |
| 同一任务被多人审批（角色审批，**单人规则**） | 第一个审批的人生效，其余人再操作返回 **已处理**（幂等） |
| 双人/顺序审批尚未完成时 | 仅当前阶段合法审批人可操作；非本阶段审批人拒绝；满额后进入 `approved` 或下一阶段 |
| IM 回调签名验证失败 | 拒绝处理，记录安全日志 |
| IM 回调重放（重复 nonce / 过期时间窗外） | 拒绝处理，记录安全日志 |
| 回调中审批人 IM 身份与任务审批人不一致 | 拒绝处理，记录安全日志 |
| 职责分离开启且创建人审批本人任务 | 拒绝通过，提示无权限 |

### 7.3 并发与幂等

| 场景 | 系统行为 |
|------|---------|
| 审批人在控制台和 IM 同时点击通过 | 后端幂等处理，仅第一次生效 |
| 运维人员重复提交相同命令 | 允许创建多个任务（每次是独立任务） |
| 审批超时与审批操作同时发生 | 以先到达的为准（乐观锁） |

### 7.4 审批委托、转派与升级（合规与体验补齐）

为满足企业合规与高可用审批体验，系统应支持以下能力（可按 P1/P2 逐步实现，但需在方案中明确）：

| 场景 | 系统行为（建议） |
|------|------------------|
| 审批人休假/不可达 | 支持 **审批委托（delegate）**：审批人可预先设置委托人（有效期 + 范围），委托操作需审计。 |
| 任务等待超过 SLA | 支持 **升级（escalation）**：达到阈值（如 15 分钟/30 分钟）后自动通知备选审批人或上级；升级规则由策略配置并写入审计。 |
| 审批人不再合规（离职/禁用） | 系统自动将任务标记为「需要重新指定审批人」或转派到管理员队列，并通知创建人。 |
| 转派审批 | 在保持职责分离的前提下，允许审批人将任务 **转派** 给同等权限的其他审批人（需理由，写入审计），防止“卡死”。 |

## 9. 审计与合规

### 9.1 审计事件类型

| 事件类型 | 说明 |
|---------|------|
| `command.task_created` | 命令任务创建 |
| `command.task_approved` | 命令任务被通过 |
| `command.task_denied` | 命令任务被拒绝 |
| `command.task_cancelled` | 命令任务被取消 |
| `command.task_expired` | 命令任务审批超时 |
| `command.execution_completed` | 命令执行完成（含结果） |
| `command.emergency_bypass` | 紧急模式跳过审批（额外记录原因） |
| `approval_policy.created` | 审批策略创建 |
| `approval_policy.updated` | 审批策略修改 |
| `approval_policy.deleted` | 审批策略删除 |
| `im_provider.created` | IM 渠道配置创建 |
| `im_provider.updated` | IM 渠道配置修改 |
| `command.risk_downgrade_requested` | 用户申请将有效风险降级（待批准） |
| `command.risk_downgrade_approved` | 风险降级已批准，任务以新低等级重算策略 |
| `command.risk_downgrade_denied` | 风险降级被拒绝 |
| `command.policy_snapshot_locked` | 任务创建时锁定审批策略快照（含 policy_id、version） |
| `command.governance_blocked` | 命令命中黑名单/危险模式被拦截 |
| `command.emergency_rate_limited` | 紧急通道触发频控阻断或降级 |
| `command.post_incident_review_missing` | 紧急执行后补录逾期 |
| `audit.evidence_export.requested` | 监管/内审 **证据包导出** 申请（`financial_gov`） |
| `audit.evidence_export.approved` | 证据包导出已批准 |
| `audit.evidence_export.completed` | 证据包已生成并完成下载/投递（不含明文密钥） |
| `compliance.profile.updated` | 租户合规配置/ `compliance_profile` 变更 |

### 9.2 紧急模式审计

紧急模式的审计记录包含额外字段：

| 字段 | 说明 |
|------|------|
| bypass_reason | 紧急原因（运维人员填写） |
| bypass_type | 跳过类型：emergency / privileged_user / auto_policy |
| approver_notified | 被通知的审批人 |
| notification_channels | 通知渠道（feishu / wechat_work / dingtalk） |

管理员可在控制台"审计事件"页面通过筛选 `event_type = command.emergency_bypass` 查看所有紧急下发记录，监控是否被滥用。

### 9.3 审计留存、导出与不可抵赖（合规补齐）

| 要求 | 说明 |
|------|------|
| **留存期（Retention）** | 命令任务、审批记录、执行结果、审计事件应支持按租户配置留存期（如 180 天/1 年/3 年）。到期支持自动清理或归档。 |
| **导出与访问控制** | 审计导出（CSV/PDF/JSON）需权限控制与二次确认；导出行为本身必须生成审计事件（如 `audit.exported`，可在 RBAC 扩展中定义）。**金融/政企证据包** 见 **§13.6**。 |
| **不可抵赖（Non-repudiation）** | 对审批动作（approve/deny/transfer/delegate）建议记录：操作者平台用户 ID、IM 身份映射、请求签名校验结果、时间戳、来源渠道（console/IM）、幂等键与请求摘要哈希；用于事后追溯与争议处理。 |
| **敏感字段分级** | 执行输出与审计字段需分级展示：默认脱敏；仅 `audit:read_sensitive` 可查看明文敏感信息；访问需审计。 |

## 12. 评审整改补充（v1.1 规范性说明）

本节汇总产品/安全评审结论，与上文章节 **一并作为实现与验收依据**。若与旧版描述冲突，**以本节为准**。

### 12.1 风险等级与服务端有效等级

- **有效等级 `effective_risk`**：由服务端根据 **命令解析结果**（关键词、正则、匹配模板、危险模式库）与 **最小等级下限**（如含 `rm -rf`、格式化磁盘、改防火墙、改关键配置路径等强制至少 L3）计算。
- **仅上调**：用户选择的 UI 等级若 **低于** 服务端计算结果，以服务端为准；用户须走 **§3.1.1** 所述 **降级申请** 才能降低等级。
- **默认需要审批（v1.1）**：系统默认 **所有任务均需审批**（含 L1、含 IM 自然语言创建），只有在以下条件同时满足时，才允许自动通过：
  - 管理员在审批策略中显式启用 `auto_approve = true`
  - `effective_risk = L1`
  - 未命中黑名单/危险模式/法定禁止命令
  - 通过职责分离等风控校验
  
  否则一律进入 `pending_approval`（或被拦截）。

### 12.2 策略快照与双人审批

- **策略快照**：任务创建时持久化 `matched_policy_id`、`policy_version`、`approval_rule`、`approver_binding`（人或角色解析结果）、`timeout_at` 等；后续策略编辑 **不改变** 已创建任务。
- **双人审批**：两名审批人须 **不同主体**；若角色审批，两次通过须来自 **不同用户**。与 **职责分离**（创建人不可批自己）同时生效。
- **顺序审批**：预审人与终审人由策略配置；前一阶段未通过不得进入下一阶段。

### 12.3 命令可控化（模板 / 白名单 / 黑名单）

- **推荐路径**：优先提供 **命令模板 + 参数化**（占位符校验、类型、枚举），减少任意 Shell。
- **黑名单**：全局/租户级禁止模式（正则）；命中则 **创建失败** 或 **强制 L3 + 双人**（由策略配置）。
- **白名单（可选）**：对特定环境（如生产）仅允许白名单内模板或前缀；任意自由文本命令禁止或需额外角色 `command:adhoc_shell`。
- **批量与标签加权**：目标设备数超过阈值或标签命中「生产」时，`effective_risk` **至少升一级** 或强制双人审批（可配置）。

### 12.4 执行输出与敏感信息

- **输出脱敏**：回传控制台/UI/IM 卡片前对已知 secret/token/密码模式做 **掩码**；完整日志可仅限 **高权限审计员** 查看（权限 `audit:read_sensitive` 或等价，可在 RBAC 扩展中定义）。
- **保留期**：执行输出与审计 **保留期限** 符合租户合规设置；导出需审计。

### 12.5 IM 审批回调安全（摘要）

与 §3.6.1 补充一致：**签名 + 时间窗 + nonce 去重 + 幂等键 + 审批人身份与任务绑定校验**；详见 §7.2 异常表。

### 12.6 紧急通道与 §3.4.7 的对应关系

临时授权、频控、事后补录、L3 强化审计等 **均以 §3.4.7 表格为产品要求**，对应审计事件见 §9.1。

## 13. 金融与政企合规增强要求

本节面向银行、证券、保险、政务及 **关基** 等行业常见监管与内控要求，作为 **验收基线（profile）** 的可选加强项。租户可通过 **`compliance_profile`**（示例取值：`standard` / `enterprise` / `financial_gov`）切换默认策略强度；**`financial_gov` 默认启用本节带「应」字条款**（实现上可拆 P1/P2，但须在 `04-design.md` 标注差距）。

### 13.1 基本原则（与人民银行、银保监会实践及等保思维对齐的方向性要求）

| 原则 | 产品要求（摘要） |
|------|------------------|
| **最小权限** | 生产变更执行权、审批权、策略配置权、审计导出权 **四权分立**；禁止同一自然人长期同时持有「申请人 + 终审人」组合（可通过岗位轮换与委托时限控制）。 |
| **双人控制 / 四眼原则** | 生产环境 **L2/L3** 默认 **至少双人或顺序审批**；紧急通道 **不得单独放宽** 双人要求，除非董事会/应急制度授权且记录在案（见 §3.4.7）。 |
| **可审计、可举证** | 任意审批、执行、导出、策略变更均可追溯到 **人、时间、终端渠道、理由、策略快照哈希**（§9.3）；监管或内检查询时可一键生成 **证据包**（§13.6）。 |
| **数据主权与传输** | 审计与工单数据 **默认境内存储**；IM 渠道选用须满足 **数据跨境/外传** 合规（政企常限定企微/钉钉 **私有化或政务云**，见 §13.5）。 |

### 13.2 变更管理与工单绑定（RFC / CAB）

| 要求 | 说明 |
|------|------|
| **变更单强制** | 对 **生产** 或 `effective_risk ≥ L2` 的任务，`financial_gov` 模式下 **变更单/工单号必填**；可与 CAB 排期窗口校验（仅允许在变更时段内执行）。 |
| **变更分类** | 表单或策略支持 **标准变更 / 紧急变更 / 例行变更** 分类；紧急变更须关联 **事件单**，并在审计中标记 `change_class=emergency`。 |
| **计划时间窗** | 可选字段 **计划开始/结束时间**；超出窗口的执行触发告警或二次确认。 |

### 13.3 审批与特权强化

| 要求 | 说明 |
|------|------|
| **阶梯认证（Step-up）** | 对 **紧急通道**、**审计导出**、**策略发布** 等高风险操作，应触发 **二次验证**（MFA/强认证），具体对接租户 IdP。 |
| **紧急通道更严** | `financial_gov` 下建议：**双人授权才能紧急**（第二人可在短时间窗口内追认）或 **紧急后 4 小时内管理层补签**；与 §3.4 并列时应取 **更严者**。 |
| **会话与证据** | 可选对接 **堡垒机/运维审计** 会话号，写入任务元数据，便于监管关联。 |

### 13.4 审计日志防篡改与留存

| 要求 | 说明 |
|------|------|
| **不可抵赖增强** | 关键审计记录建议 **仅追加（append-only）** 存储，或 **哈希链 / 定时封包** 对账；删除/修改权限仅存在于 **系统级Break-glass** 且必须 **双人+全量审计**。 |
| **留存分级** | 至少区分 **操作审计（如 1～3 年）** 与 **监管抽样包（如 3～7 年）**；支持 **法务冻结（legal hold）** 暂停淘汰。 |
| **时钟源** | 全链路时间戳建议对接 **可信 NTP / 统一时区**；IM 回调时间窗与审计时间对齐。 |

### 13.5 IM 与即时通讯治理

| 要求 | 说明 |
|------|------|
| **渠道白名单** | `financial_gov` 下由租户配置 **允许使用的 IM 类型与实例**（如仅企微政务版、钉钉专有版）；**禁止个人即时通讯**承载审批回调（可配置开关）。 |
| **外传最小化** | 审批卡片默认 **脱敏**；**完整命令与输出** 不得在 IM 明文展示，仅提供 **链接跳转控制台**（已登录鉴权后查看）。 |
| **离线审批** | 审批人必须通过 **企业统一身份** 访问控制台或经批准的 IM 集成；禁止仅凭匿名 Webhook。 |

### 13.6 监管配合与证据包导出

| 要求 | 说明 |
|------|------|
| **证据包内容** | 单次导出包含：任务元数据、策略快照、全部审批记录、执行结果摘要、相关审计事件、操作者身份映射表（脱敏规则外单独附件）。 |
| **导出审批** | 证据包导出须 **双人或安全管理员审批**（可配置），并产生 `audit.export_approved` 类事件（命名可在实现中统一）。 |
| **格式与完整性** | 支持 **带数字摘要清单**（manifest + SHA256）的压缩包，便于第三方存管或律师函证场景。 |

### 13.7 与 §12 的关系

- §12 为通用「评审整改」基线；**§13 在 `financial_gov` 下覆盖或收紧** §12 中若干「建议/可选」条款（例如：生产禁用 `auto_approve`、IM 不外传完整命令、紧急双控等）。  
- 若条款冲突，**按租户 `compliance_profile` 与监管招标文件取值**；文档未约定的以合同约定为准。
